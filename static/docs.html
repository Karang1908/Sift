What is Sift

Sift is a single-screen document analysis dashboard. A small, fixed set of hardcoded accounts log in, upload documents (PDF, DOCX, XLSX, PPTX, images, text, RTF, ODT), get them parsed to text server-side, run AI-assisted analysis queries against a local Ollama instance, and export the results as formatted Excel, PDF, or Word documents.

Every account's data is filesystem-isolated from every other account's. An is_admin-flagged account can see a permanent, append-only audit trail of everyone's activity through a separate admin panel, including copies of files, generated analysis, and exported documents that survive the original user's own deletion.

It runs as one Python process on one machine, reachable either only from that machine (127.0.0.1) or from every device on the local network (0.0.0.0). There is no cloud backend, no container orchestration, no database server — persistence is flat files on the host's own disk, the same way a small FTP server or a static site generator works. This is a deliberate scope choice: simpler operations, zero cloud cost, zero external dependencies at runtime, and the entire audit trail lives in a directory you can back up with rsync.

Stack: FastAPI + Uvicorn (Python 3.13+), vanilla JS frontend, local Ollama daemon proxying to minimax-m3:cloud. Deployment model: single-process, single-host, LAN-shared, no database, no cloud hosting.

Who It's For

Sift is sized for a small, known team on a trusted network — the original use case is a handful of people on a home or office LAN, each with their own private workspace, sharing one machine. It is not a public-facing SaaS: there is no signup flow, no rate-limiting beyond a per-username lockout, and no transport encryption (the session cookie and login credentials travel in plaintext on whatever network the server is bound to). See the Known Limitations section for the full disclosure, and the Security Model section for what is actually enforced and where the trust boundary lives.

Features

Multi-Format Parsing

Upload PDF, DOCX, XLSX, PPTX, images (via OCR), RTF, ODT, and plain text. All files are parsed server-side into cached text for instant analysis.

AI-Powered Analysis

Run natural-language queries against your uploaded documents via a local Ollama instance. Results stream in real-time via Server-Sent Events.

Formatted Export

Export analysis results to Excel, PDF, or Word. Two pipelines: an AI script generator and a template-based clone pipeline for precise formatting.

Per-User Isolation

Every user's uploads, caches, presets, and templates live in their own filesystem namespace. No shared pools, no cross-user data leaks.

Permanent Audit Trail

Every upload, analysis, and export is archived permanently. Deleting from the workspace never touches the archive. Admins see everything.

Prompt Enhancement

Refine your analysis prompts and export instructions with one-click AI rewriting before running them — better input, better output.

Presets System

Save and reuse frequently-used analysis prompts and export instructions as named presets. Separate systems for analysis and export workflows.

One-Click Launcher

Double-click start_mac.command or start_windows.bat. Handles Ollama, dependencies, the server, and opens your browser.

Component Architecture

Everything inside Host Machine is one OS process. There is no internal network hop between the web layer, the parser, the audit log, and the filesystem — those are plain Python function calls inside the same address space. The only two real network hops are Browser ↔ Server (LAN, plaintext HTTP) and Server ↔ Ollama (loopback, then out to the internet for :cloud-suffixed inference).

What lives in each box

  • Web layer (app.py, ~2,785 lines): every HTTP route, the bcrypt-hashed user database, the in-memory session table, all the per-user path helpers, and both export pipelines. This is the entire server.
  • Parser (parser_utils.py, 206 lines): one function, extract_text_from_file(), that dispatches by file extension to the right library (pypdf for PDF, python-docx for DOCX, openpyxl/xlrd for spreadsheets, python-pptx for slides, pytesseract for OCR, macOS textutil for legacy formats). Always returns text — on failure it returns a bracketed sentinel string that the caller can detect.
  • Audit log (audit_log.py, 233 lines): owns the logs/activity.jsonl append-only event log plus the three permanent per-user archive directories. Imported by app.py, never imported the other way — the web layer is the only writer.
  • Storage: eight top-level directories, all flat files, no SQLite, no Redis, no anything. The per-user isolation guarantee is structural (each account's data lives under a different directory that the authenticated request handler joins into paths), not enforced by any access-control check.
  • Ollama daemon: a separate process (ollama serve) listening on 127.0.0.1:11434. Sift does not embed a model — it just makes HTTP calls. The :cloud model is actually Ollama's own daemon proxying inference to Ollama Cloud, so the user needs to have ollama login on this machine once.

In this codebase these are not gated by get_current_user — anyone who can reach port 8000 can browse the full API shape (every route, every parameter name). No actual data is exposed through them (they describe the API, they don't call it), but it is real unauthenticated surface worth knowing about. See API Surface for the full table including which endpoints are gated and which are not.

Authentication & Sessions

Sift uses session cookies with a server-side in-memory session table — deliberately not JWT. A JWT was considered and rejected for this codebase: the deployment model is a single-process local tool, and an in-memory store gives instant revocation (a click of "Logout" actually invalidates the session) with no signing key to manage. Sessions are fixed-TTL from login time, not sliding — a user active at hour 5:59 is still logged out at hour 6:00.

Design choices worth understanding

  • Why bcrypt with a dummy hash for unknown usernames: bcrypt is intentionally slow (~100ms per check). Without the dummy-hash trick, an attacker could time login responses to enumerate which usernames exist (instant reject for an unknown user, 100ms for a real one). The dummy hash, computed once at import time, makes the timing constant whether the username is real or not.
  • Why the entire team logs out on every restart: the session table is in-memory only. Any uvicorn --reload triggered by editing a watched file during development wipes every active session. This is a deliberate simplicity tradeoff (no session persistence to disk, no signing key to manage), documented in CLAUDE.md as a known behavior — not a bug. If mid-edit kicks everyone off, the cost is "everyone logs in again".
  • Why /api/logout doesn't require a valid session: an already-expired cookie should still be able to clear its own server-side entry and the browser-side cookie. A 401-on-logout would mean stale sessions pile up forever in the session table until the server restarts.
  • Why the cookie is HttpOnly + SameSite=Lax: HttpOnly prevents client-side JavaScript from reading the token (so a hypothetical XSS in a future feature can't steal the session). SameSite=Lax prevents the cookie from being sent on cross-site POST navigations — combined with the wildcard CORS but no credentials, the cookie can't be ridden by a third-party site.

Per-User Data Isolation

Every account's data lives under its own subdirectory (or its own named file, for the two preset stores), never a shared pool. The username value used in any path is only ever the return value of get_current_user() — a validated USERS key, never raw request input — so joining it directly into a filesystem path is safe. The isolation guarantee is therefore structural, not just permission-checked: there is no shared directory another account could even attempt to escape into.

Why "structural" matters more than a permission check

The alternative design would be a single shared uploads/ directory with each file's owning user recorded in a sidecar file or database, and a permission-check helper called on every read and write. That works, but it has the shape of a class of bugs: a single forgotten if owner == user check in some new endpoint leaks another user's file. With structural isolation, there is nothing to forget — the account named in the path is the only path that exists for that account. The bug class is gone, not just every individual instance of the bug.

Two independent isolation mechanisms are stacked:

  1. Live workspace (uploads/, parsed_cache/, presets/, export_templates/, export_presets/) — reflects exactly what a user currently sees in their own UI. Deleting a file removes it from here. The two preset stores use a single JSON file per user rather than a directory (the schema is a flat dict, no need for nesting).
  2. Permanent archive (audit_uploads/, audit_analysis/, audit_exports/) — written once at the moment content is created (upload time / generation time), never modified by later user actions. Each entry is a content file plus a .meta.json sidecar with the timestamp, original filename, and any format-specific metadata (export instructions, AI vs. deterministic path, etc.). The admin panel reads these directly.
  3. Activity log (logs/activity.jsonl) — the one genuinely shared file in the whole system, by design (it's the admin's cross-user activity feed). It is write-only from each request's perspective (a single line appended under a threading.Lock) and read-only from the admin panel's, so the shared file is not a concurrency or isolation risk.

Why double-store uploads?

A file lives in uploads/<user>/ while the user can see it, and an immutable copy lives in audit_uploads/<user>/ from the moment it was uploaded. The cost is approximately 2× disk for live files. The benefit is that deleting a file from the live workspace cannot lose the archive, and there is exactly one write path to get right (upload time) rather than two (upload and delete, with the delete path having to survive partial failures). This is the same shape as email systems that keep a "sent items" copy even after the recipient deletes the message — a deliberate "never lose an actual thing" tradeoff.

Request Flow: File Upload

A single POST /api/upload call triggers five side effects in sequence: write the original, parse it, cache the extracted text, archive the original, and log the activity. The four side effects after the first are skipped if any earlier step throws — the whole upload is atomic from the caller's perspective.

    P-->>S: parsed text (or bracketed Error/Unsupported sentinel)
    S->>FS: write <file>.txt -> parsed_cache/<user>/ (atomic replace)
    S->>AL: archive_upload_copy() -> permanent copy in audit_uploads/<user>/
    S->>AL: log_activity(user, "upload", filename, size, status, record_id)
    S-->>B: 200 {filename, size, status: "parsed" | "error"}

Why the "sentinel string" pattern matters

Every failure path in extract_text_from_file() appends a [Error ...] or [Unsupported ...] string as the entire cache content rather than raising. The reason: the cache file always exists after upload, so the listing endpoint has to be able to tell "this file parsed successfully" apart from "this file's cache only contains a placeholder". A separate status file, a sidecar, or a sentinel-on-raise approach all introduce a path where a partial failure can be silently indistinguishable from a success. The bracketed sentinel is checked by parser_utils.is_error_content() in the file listing endpoint, which reports status: "error" instead of status: "parsed" for any cache whose first 200 chars start with one of the sentinel prefixes.

The permanent archive copy is made at upload time, independent of whatever happens next. A delete two seconds later cannot lose it. A crash in the middle of the cache write cannot lose it (the archive write is its own line item in the sequence, after the cache write succeeds).

Request Flow: Analysis Query (SSE)

The /api/process endpoint is the core of the app. It concatenates every parsed-text file in the current user's parsed_cache/ directory into a citation-tagged context block, then streams a completion from Ollama as Server-Sent Events. Before the audit-trail feature, the generated analysis was assembled in the browser and discarded server-side the moment the stream ended — now it is also persisted permanently.

    S->>S: "concatenate every .txt in user's parsed_cache<br/>(citation-tagged: --- START OF DOCUMENT: filename --- ... --- END ---)"
    S->>O: POST /api/chat {model, messages, stream:true}

        S-->>B: SSE: data: {"content":"...", "done":false}
        Note over S: accumulate content in memory
    end
    O-->>S: {done: true}
    S-->>B: SSE: data: {"content":"", "done":true}
    deactivate S
    S->>AL: "asyncio.shield(persist task):<br/>archive_analysis() then log_activity(\"process\")"
    Note over AL: "shielded so a client disconnect right<br/>at stream-end can't cancel the write<br/>between the two steps"

Why asyncio.shield() on the persist task

StreamingResponse will cancel the inner event-generator task as soon as the client disconnects. Without shielding, a client that closes the browser tab at exactly the wrong moment can cancel the request task between the archive_analysis() call and the log_activity("process") call — leaving an archived .md on disk with no matching activity-log entry, an inconsistent audit trail. The shield wraps the persist task so the cancellation lands on the outer request task but the background persist runs to completion either way.

This was found live during the audit-trail verification run before the fix, and the _persist_process_archive helper was extracted so the same shield pattern is the single source of truth (also applied to _stream_export() in the next section).

Pipeline selection rule

The clone pipeline is used when the template has discrete, mappable structure: an Excel workbook with cell coordinates, a Word document with paragraph locations, or a PDF with real AcroForm fields the model can fill. The script pipeline is used when there is no template, or when the template is a fixed-layout PDF (a brochure layout, a report template with no form fields) where the only way to "fill" it is to overlay text on top of the existing artwork.

Why the clone pipeline is the safer default

The clone pipeline is the meaningfully smaller attack surface: the model never writes or runs code. It only returns a bounded JSON object ({"<location_id>": "<value>"}) that a trusted, backend-authored splice function applies to a copy of the template. Even if the model returns malicious JSON, the worst it can do is fill in the wrong values — it cannot escape the mapping into a code execution context.

The script pipeline does run LLM-generated Python on the host. It is guarded by:

  • An AST import/name allowlist per format (e.g. only openpyxl is allowed for Excel scripts, only reportlab + pypdf for PDF, only python-docx for Word). Any other import or name access fails validation.
  • A subprocess with a CPU-time rlimit and a wall-clock timeout. The script is killed if it tries to take forever or peg the CPU.
  • A deterministic fallback if the script fails twice in a row — _generate_*_bytes() uses a hand-rolled, never-model- generated parser to produce a reasonable output from the same markdown input.

This is defense-in-depth, not a real sandbox. There is no seccomp, no chroot, no network namespace. It's appropriate for a local single-trust-boundary tool; it would not be appropriate to port to a multi-tenant or externally-reachable deployment without adding real sandboxing first. The deterministic fallback is the actual safety net for both pipelines — when model-written code is doing something unexpected, the user still gets a valid file out of it.

Real bug caught in live testing

For .docx templates, a label and its placeholder often share a single text run (e.g. "Solar Conversion Rate: [VALUE]"). The original naive implementation overwrote only the placeholder substring, destroying the label. The fixed model prompt and the splice function now require the model to return the complete replacement text for any label-and-placeholder run, not just the bare value. This was found and fixed during the original export-pipeline work and is documented in .agents/skills/excel_export/SKILL.md.

Streaming progress to the browser

Both pipelines push real per-stage progress over the same SSE channel. The clone pipeline's stages are template → schema → mapping → splice → done. The script pipeline's stages are prompt → generate → validate → execute → fallback → done. The progress bar in the UI moves with the real pipeline rather than a single static "generating..." message. The terminal SSE event carries the file as file_b64 plus a filename and mime type, and the browser decodes it client-side to trigger the download.

The one property you should never lose

Deleting a file from the live workspace never touches the archive. delete_file() removes from uploads/<user>/ and parsed_cache/<user>/ only. The archive copy made at upload time stays where it is, and remains admin-visible/downloadable indefinitely. There is currently no retention or pruning policy on the archive — it only ever grows, a deliberate tradeoff of "never lose an actual artifact" over disk efficiency for a small team.

The admin panel

The admin panel is a standalone page at /static/admin.html, served by the same blanket StaticFiles mount that serves the main app — no extra routing. It shares the style.css palette and class system for visual consistency, but does not share any JS/DOM state with script.js. On page load it calls /api/me; if is_admin is not set on the logged-in account, it shows an access-denied state instead of the panel. Otherwise:

  • Users tab — the configured accounts from app.py's USERS dict, with admin badges.
  • Activity tab — the activity.jsonl feed, filterable by username and action, newest first. Every event in the system shows here.
  • Uploads tab — every archived upload, with a live / deleted status badge and a download link. Critical property: a file the user deleted from their own view still shows up here and is still downloadable.
  • Analysis Runs tab — every archived analysis run, with the prompt length, document count, and a complete / incomplete status. Clicking View opens a modal with the full generated markdown rendered as the model produced it.
  • Exports tab — every archived exported document, with format and source (AI vs deterministic). Download link serves the original file bytes.
  • Presets & Templates tab — a flat list of every saved analysis and export preset across all users, with attached template downloads for the export presets.

The still_live flag, defined correctly

A subtle bug found and fixed during verification: the original still_live flag was computed per-filename-existence. If a user deleted a file and re-uploaded one with the same name, the old archive record would show still_live: true because a file with that name now exists, even though those bytes belong to the newer record, not the older one. The fix: only the newest archive record for a given (user, filename) pair can ever be still_live. An older revision superseded by a later upload of the same filename correctly shows as not-still-live, even when a file with that name exists again.

Security Model

Sift's threat model is: trusted LAN, small known team, single host. Every control here matches that model. None of this is secret, and the limitations are documented honestly in the next section rather than glossed over.

The trust boundary, stated precisely

The single most important sentence in this document:

This is correctly scoped as a trusted-LAN, known-small-team tool. Every control above matches that threat model. The single biggest change that would be required to expose this beyond a trusted LAN is transport security — put a reverse proxy with a real TLS certificate in front of it. Nothing inside the app itself provides encryption, because the deployment model is a private network with known participants, not a public service.

Known Limitations

Things this codebase deliberately does not do, and what to do if you need them.

  • No TLS / HTTPS. Login credentials and the session cookie travel in plaintext on whatever network the server is bound to (127.0.0.1 = safe, 0.0.0.0 on a LAN = sniffable by anyone else on that broadcast domain). For a home network or a small trusted office LAN this is fine; for public WiFi it is not.
    Fix if needed: put a reverse proxy (Caddy, nginx) with a TLS certificate in front of this app. Nothing inside app.py needs to change.
  • No password reset, no signup, no email. Accounts are hardcoded in app.py's USERS dict. To add an account: generate a bcrypt hash, add the entry, restart. There is no flow for the account owner to recover a forgotten password on their own — the operator of the host machine has to do it.
    Fix if needed: the existing USERS dict is the right shape; add a small admin-only endpoint that takes a username and a new password and writes a fresh hash.
  • No archive / activity-log retention policy. The audit archive and activity.jsonl only ever grow. For a small team over a multi-year horizon this is unbounded disk growth.
    Fix if needed: a one-off pruning script that deletes audit_uploads/<u>/<record>__file entries older than N days, with the matching log line and any referenced audit_exports/<record>__file and audit_analysis/<record>.md. Deliberately not built in because for a small team, "never lose an actual artifact" outweighs the disk cost.
  • LLM quota is shared across all accounts. The :cloud model proxies through a single Ollama account with ~5M tokens/week. Multiple Sift accounts now share that quota, unlike the single-recipient assumption from before the multi-account rework.
    Operational impact: heavy admin-panel verification runs (or really any heavy use by any account) can exhaust the weekly quota outright. test_backend.py or any real /api/process call will start returning HTTP 429 ("you have reached your weekly usage limit") mid-session, not from any code regression. If you see that, check the response body before assuming a bug, and don't burn more quota re-running live checks to "confirm" it.
  • No real sandbox around LLM-written Python. The script pipeline's AST allowlist and subprocess rlimits are defense-in-depth, not a real sandbox. Acceptable at the local single-trust-boundary; would need seccomp / chroot / network namespace before porting to a multi-tenant or externally-reachable deployment.
  • No code-signing certificate. Any PyInstaller-built executable will trigger an unsigned-binary warning on first launch (Gatekeeper on macOS, SmartScreen on Windows). The launcher scripts hit the same class of warning.
    Fix if needed: an Apple Developer account for macOS notarization, an EV code-signing cert for Windows. Out of scope for this codebase.
