Sift · Document intelligence

Understand how Sift works.

Sift ingests documents, extracts their text, answers questions about them with an AI model, and exports the result as a formatted PDF, Excel workbook or Word document. This page documents the entire system end to end — every layer, every request flow, every route, and every deliberate trade-off.

What Sift is

Sift is a single-screen dashboard. Everything happens on one page: you upload documents on the left, write a question below them, and the answer streams into the panel on the right, from which you export it.

The mental model is deliberately simple. Sift maintains, per user, a working set of documents. Every question you ask is answered against the entire working set — not a selection, not a search result, but all of it. Adding a document widens what the model can see; deleting one narrows it. There is no index to rebuild and no "select which files to query" step, because there is no retrieval layer at all.

It is a local / LAN tool for a small, known team, not a public web app. There is no sign-up, no password reset and no database: accounts are hard-coded in source and all state is plain files on disk.

What it is not

  • Not a RAG system. No embeddings, no vector store, no chunk ranking. See Analysis for why that is a deliberate choice and what it costs.
  • Not multi-tenant in the SaaS sense. Accounts are isolated from each other, but every account is someone you personally trust and added by editing source.
  • Not internet-facing. There is no TLS, and one export path executes model-written code locally. Both are safe under the intended threat model and unsafe outside it.

Network & trust boundary

Three participants. The browser never contacts the model directly, so the API key stays on the server and no document content is exposed to the client beyond what you already uploaded.

Fig. 2 — The dashed boundary is the only thing protecting the plaintext hop on the left.

The left hop is unencrypted Browser → server is plain HTTP. On 127.0.0.1 that traffic never leaves the machine and is fine. Bound to 0.0.0.0 for the team, credentials and the session cookie cross the network in clear text. The server → model hop is HTTPS regardless.

Authentication & sessions

Accounts live in a USERS dictionary in app.py. Each entry holds a bcrypt hash — never a plaintext password. Storing a bcrypt hash in source is intentional and safe: it is designed to be public-readable, unlike an API key.

Fig. 3 — Login, the gate, and the two ways a session ends.

Properties worth knowing

  • Fixed expiry, not idle-based. The session dies six hours after login, however actively you were using it.
  • Sessions are in memory. Restarting the server signs everyone out — including an auto-reload triggered by editing a watched file in development.
  • Username enumeration is blocked. An unknown username is still checked against a dummy hash, so a wrong username and a wrong password take the same time.
  • Logout is forgiving. It does not itself require a valid session — an already-expired cookie can still log out cleanly.
  • Admin is a flag, not a role system. An account marked admin keeps using the app exactly as before and additionally reaches the panel.
401 handling is centralised Every authenticated call goes through one wrapper on the frontend. Any 401 re-shows the sign-in screen and clears in-memory state — the report, the export buttons, presets and template selections — so a re-login (possibly as a different person) can never reveal or export the previous session's work.

Per-user isolation

Isolation is structural, not permission-checked. Each user's data lives under a path built from their username, so there is no shared location for one account to reach into — the wrong path simply does not exist.

Fig. 4 — One validated username fans out to every storage location.

Why joining a username into a path is safe here The username is never taken from request input. It always comes out of the session gate, which returns a key that already exists in USERS — so it cannot contain ../ or anything else path-hostile. Admin routes are the one exception: they accept a username as a parameter, and therefore validate it against the account list before it is used to build a path.

Request flow — upload & parse

Uploading is synchronous from the user's point of view: by the time the row appears in the ledger, the document has already been parsed and cached, and its permanent archive copy exists.

Fig. 5 — Upload pipeline. Step 06 is why deleting a file never loses the audit copy.

Response shape

{
  "filename":     "Q3_report.pdf",
  "size":         1284910,     // bytes of the original upload
  "status":       "parsed",    // or "error" if extraction produced only a sentinel
  "parsed_chars": 48213        // size of the extracted text — drives the context gauge
}
Why parsed_chars and not size They are unrelated. A 10 MB scanned PDF may extract to a few hundred characters; a small image may OCR into thousands. Only the extracted text is sent to the model, so only parsed_chars predicts context usage.

Failure behaviour

Extraction never raises. A failing branch returns a bracketed sentinel — [Error parsing PDF: …] — which is written to the cache like any other text. The upload endpoint recognises those sentinels and reports the file with status error instead of parsed. Consequences worth understanding:

  • One unreadable file never aborts a batch of uploads.
  • An errored file still occupies a (tiny) slice of context, because its sentinel text is included like any other document.
  • Adding a new failure message means also registering its prefix, or the file will misreport as successfully parsed.

Why there is no retrieval

Sift does not embed, index, chunk or rank. It concatenates. That is the naive baseline that RAG exists to replace, and for this tool it is usually the better trade:

Sift (full context)RAG (retrieve then generate)
Model sees100% of every documentOnly the top-k retrieved chunks
Fails whenTotal text exceeds the windowRetrieval selects the wrong chunks
Cost / latencyScales with the whole corpus, every queryScales with the retrieved slice
ComplexityA loop and a string joinVector store, embeddings, chunking, indexing
AttributionWhole document present and filename-taggedChunk-level; surrounding context can be lost

For a handful of business documents, full context is more accurate than RAG — there is no retrieval step to get wrong. The ceiling is the context window; see below.

Citations — what is and is not guaranteed

Each document arrives wrapped in markers carrying its filename, and the system prompt asks for quotes and specific references. So:

  • Document-level attribution is reliable — the filename is structurally present next to the text it labels.
  • Finer citations are best-effort. Page, slide and row numbers are only as good as what survived extraction; the parser does not inject page markers, so such references can be approximate.
  • Nothing is machine-verified. Citations are prose the model writes, not a checked data structure.

The context window & token gauge

Because the whole working set is sent every time, the context window is the binding constraint. The file panel therefore shows a live gauge.

ZoneRangeColourWhat to do
Normal< 75%Arc blueNothing — comfortable headroom.
Caution75 – 89%AmberConsider removing documents you are not asking about.
Danger≥ 90%RedTrim the working set; truncation or failure is likely.

How the number is produced

tokens  ≈ sum(parsed_chars for every file) / 4
percent =  tokens / 1,000,000 × 100

Roughly four characters per token is the standard English approximation. It is an awareness signal, not a billing meter — shipping a full tokenizer to drive a progress bar would be disproportionate. Expect it to be in the right neighbourhood, not exact, and to drift furthest on non-English or heavily symbolic text.

1M vs 512K — read this before trusting the gauge The gauge counts against 1,000,000 tokens, MiniMax M3's full advertised window. Ollama lists the minimax-m3:cloud variant at 512K, so the real ceiling here may be half what the gauge shows. Two softer limits usually bite first: input and output share the same window (fill it with documents and there is no room to answer), and the model account has a weekly token quota. Attention also degrades over very long contexts — a fact buried in the middle of an enormous prompt is likelier to be missed. "It fits" is not the same as "it is reasoned over evenly".

Export — two pipelines

Export turns the finished analysis into a real file. Which pipeline runs is decided by one question: did you attach a template?

Fig. 7 — One decision, two very different security profiles.

Pipeline 1 — generation

With no template (or a fixed-layout PDF), the model writes an actual Python script tailored to this specific report, which is validated and executed to produce the file. Guards:

  • AST allow-list — the script is parsed and rejected if it imports or references anything outside a per-format permitted set (a spreadsheet library for Excel, a PDF library for PDF, and so on).
  • Subprocess isolation — it runs in a separate process, never in the server process.
  • CPU and wall-clock limits — a runaway script is killed.
This is defence in depth, not a sandbox There is no seccomp, no chroot and no network namespace. A sufficiently creative payload that stays inside the allow-list still executes locally. This is acceptable only because Sift runs on your own machine, inside the same trust boundary as you. Do not expose this pipeline to untrusted users or the public internet without adding real sandboxing first.

Pipeline 2 — cloning

When you attach an .xlsx, .docx or a PDF with real form fields, no code is generated at all. The template's true structure is read and each writable location gets a stable id; the model is asked for only a JSON object mapping ids to values; trusted application code applies that mapping to a copy of your template.

This is both safer and more faithful — your layout, styling and formulas survive because the file is never regenerated, only filled.

A real bug this design surfaced In Word documents a label and its placeholder often share one text run — for example "Conversion rate: [VALUE]". Because the splice replaces a whole run, the model must return the complete replacement text including the label, or the label is destroyed. Any new template format should follow the mapping-and-splice pattern rather than reverting to code generation.

Progress reporting

Both pipelines stream real per-stage progress over SSE, so the progress bar reflects the actual pipeline rather than a fixed animation. The terminal event carries the finished file as base64 together with its filename and MIME type.

PipelineStages
Scriptprompt → generate → validate → execute → (fallback) → done
Clonetemplate → schema → mapping → splice → done

Markdown export is the exception: it is produced entirely in the browser from the text already on screen, needs no model call, and works offline.

Audit trail & admin panel

Two independent records, both permanent and both admin-only.

Fig. 8 — Capture points and the one-way flow into the admin panel.

Design decisions

  • Archive at creation, not deletion. One write path to get right; a crash during delete can never lose the copy. The cost is that uploads are stored twice by design.
  • Partial analyses are kept. If the browser disconnects mid-stream, whatever was generated is still archived and flagged incomplete.
  • Exports were previously never on disk — they were streamed straight to the browser. The archive is the only durable copy.
  • JSONL over a database. Grep-able, append-only, no new dependency, and trivially fast to filter at this scale.
Two things to tell your users First, delete is local to their view — the audit copy survives. Second, the archive has no retention policy and only grows. Pruning it is a deliberate manual decision, never something the app does quietly.

Deployment & configuration

The intended path is the double-click launcher — start_mac.command or start_windows.bat — which starts the server and opens the browser. Both are idempotent: running one again when everything is already up is safe.

# install dependencies
python3 -m pip install -r requirements.txt --break-system-packages

# run the server
python3 -m uvicorn app:app --host 127.0.0.1 --port 8000 --reload

Configuration lives in a local .env (see .env.example), which must stay uncommitted. Paths are resolved relative to the application directory, so the folder can be copied or moved and still work.

Adding or changing an account

  1. Generate a bcrypt hash for the new password.
  2. Add an entry to USERS in app.py (add is_admin if needed).
  3. Restart the server — which signs everyone else out, since sessions are in memory.

Only the hash belongs in the file. Never put the plaintext password in source, a commit message, or a chat.

Binding beyond localhost 127.0.0.1 keeps Sift on this machine. Serving the team means binding 0.0.0.0, which — with no TLS — puts credentials on the network in clear text. Understand that trade-off before changing the bind address.

Security model

The threat model is a small group of known, trusted colleagues on a private network. Inside it Sift is careful. Outside it, it is not hardened — and the gaps are deliberate and disclosed rather than accidental.

AreaPosition
Passwordsbcrypt hashes only; never stored, logged or transmitted in plaintext by the server.
SessionsOpaque random tokens, HttpOnly + SameSite=Lax, 6-hour fixed expiry, server-side table.
Brute forcePer-username lockout after repeated failures; constant-time behaviour for unknown usernames.
IsolationPer-user directories keyed by the validated session username.
Path safetyUpload and template filenames normalised; admin-supplied usernames validated against the account list before use.
Rendered outputModel markdown is sanitised before insertion and fails closed to plain text if the sanitiser is unavailable — the model can echo hostile content from an uploaded document.
CORSNo credentialed wildcard. The frontend is same-origin and never relied on CORS.
TransportNone. Plain HTTP.
Generated codeAST allow-list, subprocess, CPU and wall-clock limits. Defence in depth, explicitly not a sandbox.

Known limitations

  • No TLS. Anyone on the same network can read credentials and the session cookie in transit.
  • Sessions are in memory. Any restart — including a dev auto-reload — signs everyone out.
  • Every document goes into every query. No retrieval, so a large working set costs more and can exceed the window.
  • Token counts are estimated at ~4 characters per token, not measured with the model's tokenizer.
  • The gauge assumes 1M, while the deployed variant may expose 512K.
  • The audit archive never shrinks. No retention policy exists.
  • Some legacy formats are macOS-only, and .ppt is unsupported everywhere.
  • Scanned PDFs may extract almost nothing if they carry no text layer.
  • The model account has a weekly quota. Exhausting it makes AI endpoints fail until it resets — a quota condition, not a bug.
  • Accounts are code. Adding a user means editing source and restarting.
  • One export path executes model-written code locally, under limits that are not a true sandbox.

Troubleshooting

SymptomMost likely cause
Signed out unexpectedlyThe 6-hour session expired, or the server restarted. Both are expected.
AI endpoints return 503OLLAMA_API_KEY is missing from .env.
AI calls suddenly fail after workingThe weekly token quota is exhausted. Wait for the reset window rather than retrying.
A file shows status errorExtraction produced only a sentinel — a scanned PDF with no text layer, a .ppt, or a macOS-only format on another platform.
Analysis ignores a documentCheck it is parsed, not error or pending; and check the context gauge — a very full window degrades recall.
Export produced a plain-looking fileBoth pipeline attempts failed and the deterministic fallback generator ran.
A template's layout was not preservedThe file must be a real .xlsx/.docx or a PDF with actual form fields; a fixed-layout PDF takes the script pipeline instead.
Copy button does nothingClipboard access is restricted outside a secure context; the app falls back automatically, but browser policy can still block it.
Everyone was signed out mid-workSomeone edited a watched file while the dev server ran with --reload.
